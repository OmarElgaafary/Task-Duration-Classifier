> Predict whether a Jira-style software issue is likely to be **Short**, **Standard**, or **Long-running** from its summary, description, priority, and issue type.

## Project Goal

Task Time Predictor is a full-stack machine learning project for estimating the duration category of software issues. The project includes:

- data exploration, cleaning, and feature engineering notebooks
- a scikit-learn Logistic Regression training pipeline
- a FastAPI prediction API
- a React/Vite frontend for trying predictions and viewing model metrics
- Docker support for API deployment
- Vercel configuration for frontend hosting with API rewrites
- GitHub Actions for Python linting and README metric updates

The model predicts an ordinal category rather than an exact number of days:

```text
Short -> Standard -> Long-running
```

Because the classes are ordered, a one-step miss can still be useful for planning. For example, `Short` predicted as `Standard` is less severe than `Short` predicted as `Long-running`.

## Workflow

1. **01 - Exploratory Data Analysis**
   - Inspects raw Jira fields, missing values, timestamp quality, text length patterns, categorical distributions, and duration spread.

2. **02 - Data Cleaning**
   - Loads selected raw Jira fields.
   - Keeps completed issues with valid created and resolution timestamps.
   - Normalizes text and categorical fields.
   - Creates simple counts and flags.
   - Exports `data/processed/jira_issues_cleaned.csv`.

3. **03 - Feature Engineering**
   - Calculates `duration_days`.
   - Converts durations into `Short`, `Standard`, and `Long-running`.
   - Removes noisy class-boundary records.
   - Keeps consistent project/issue-type groups.
   - Caps project/class dominance.
   - Balances the target classes.
   - Exports `data/processed/final_cleaned.csv`.

4. **Model Training**
   - `training/model_training.py` trains a scikit-learn pipeline.
   - The pipeline combines TF-IDF text features, one-hot categorical features, and scaled numeric count features.
   - Evaluation artifacts are saved to `model_tests/`.
   - The trained model is saved to `models/duration_logistic_regression_classifier.joblib`.

5. **API and Frontend**
   - `api/main.py` loads the saved model and exposes prediction endpoints.
   - `ui/` provides a React/Vite interface for trying the model, viewing API status, and displaying metrics.

## Prediction Flow

The current prediction path is API-driven. A user submits task details through the frontend or directly to `POST /predict`.

Input fields:

- `summary`: short task title or issue summary
- `description`: optional longer issue details
- `issuetype_name`: optional Jira issue type
- `priority_name`: optional Jira priority

At request time, the API creates a one-row pandas dataframe. It builds `total_text` from summary plus description, calculates text length features, fills optional metadata defaults, and sends the row into the trained scikit-learn pipeline.

The trained model currently uses these core features:

- `total_text`
- `priority_name`
- `issuetype_name`
- `summary_char_count`
- `summary_word_count`
- `description_char_count`
- `description_word_count`

The API response now includes both the predicted class and model probabilities:

```json
{
  "duration_category": "Standard",
  "probabilities": {
    "Long-running": 0.18,
    "Short": 0.27,
    "Standard": 0.55
  }
}
```

The frontend uses those probabilities to show a result chart after each prediction.

## API

FastAPI app: `api/main.py`

Available endpoints:

```text
GET  /              API metadata
GET  /health        API metadata and running status
POST /predict       Duration category prediction
GET  /docs          Interactive FastAPI docs
GET  /openapi.json  OpenAPI schema
```

Example request:

```json
{
  "summary": "Fix CORS error on /api/v2/users endpoint",
  "description": "Frontend requests are failing in production because the response is missing an Access-Control-Allow-Origin header.",
  "issuetype_name": "Bug",
  "priority_name": "High"
}
```

## Frontend

The `ui/` app is built with React, Vite, TypeScript, Tailwind CSS, lucide-react icons, and Recharts.

It includes:

- a model overview section
- API status loading from `/health`
- metrics and confusion matrix display
- a prediction form connected to `/predict`
- probability visualization for model output
- light/dark theme support

Local frontend commands:

```powershell
cd ui
npm ci
npm run dev
```

During local development, the frontend defaults to `http://localhost:8000` for API calls. A custom API base can be set with `VITE_API_BASE_URL`.

## Installation

Clone the repository:

```powershell
git clone https://github.com/OmarElgaafary/Task-Duration-Classifier
cd Task-Duration-Classifier
```

Install Python dependencies:

```powershell
pip install -r requirements.txt
```

Run the API locally:

```powershell
python -m uvicorn api.main:app --host 0.0.0.0 --port 8000
```

Open:

```text
http://localhost:8000/docs
```

## Docker

The Dockerfile builds a Python API image, installs `requirements.txt`, copies the repository, exposes port `8000`, and starts Uvicorn.

Build the image:

```powershell
docker build -t task-duration-classifier .
```

Run the API container:

```powershell
docker run --rm -p 8000:8000 task-duration-classifier
```

Then open:

```text
http://localhost:8000/docs
```

Large raw and generated datasets are excluded from the Docker build context through `.dockerignore`.

## GitHub Actions

The repository includes two automation files:

- `.github/workflows/python-app.yml`
  - runs on pushes to all branches and pull requests to `master`
  - sets up Python 3.14
  - installs dependencies
  - runs flake8 syntax and style checks
  - updates the README accuracy value from `model_tests/classification_report.txt` on push

- `.github/dependabot.yml`
  - checks Python dependency updates weekly for the root package ecosystem
